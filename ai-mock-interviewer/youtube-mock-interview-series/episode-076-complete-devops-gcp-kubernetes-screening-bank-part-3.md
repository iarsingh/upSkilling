## Question 11

Interviewer:
Explain Ingress.

Pause the video and answer this question aloud.

Senior Associate answer:
Ingress in Kubernetes is a resource that manages external HTTP/HTTPS access to services inside a cluster, typically by routing traffic based on rules like hostnames and paths.
________________


Core idea
Ingress acts like a smart traffic router (Layer 7) that controls how external requests reach internal Services.
Instead of exposing every Service with a LoadBalancer, Ingress provides a single entry point.
________________


Why Ingress is used
Without Ingress:
      * Each service needs its own external IP or LoadBalancer
      * Expensive and hard to manage
With Ingress:
      * One external entry point
      * Routes traffic to multiple services
      * Based on rules like domain or URL path
________________


What Ingress does
      * Routes HTTP/HTTPS traffic
      * Supports host-based routing
      * Supports path-based routing
      * Handles SSL/TLS termination
      * Centralizes external access management
________________


Important: Ingress vs Ingress Controller
1. Ingress (resource)
      * Just a set of routing rules
2. Ingress Controller
      * Actual component that implements those rules
      * Examples:
      * NGINX Ingress Controller
      * Traefik
      * HAProxy
      * AWS ALB Ingress Controller
Without a controller, Ingress does nothing.
________________


Example use case
You have:
      * frontend-service
      * backend-service
You want:
      * example.com/ → frontend
      * example.com/api → backend
________________


Example Ingress YAML
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
spec:
  rules:
  - host: example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: backend-service
            port:
              number: 80
________________


How traffic flows
      1. User hits example.com
      2. Request reaches Ingress Controller
      3. Controller checks Ingress rules
      4. Routes traffic to correct Service
      5. Service forwards to Pods
________________


Key features
1. Host-based routing
      * app1.example.com
      * app2.example.com
2. Path-based routing
      * /api → backend
      * / → frontend
3. TLS termination
      * Handles HTTPS certificates
      * Offloads encryption from services
________________


Ingress vs Service
Feature
	Service
	Ingress
	Layer
	L4 (TCP/UDP)
	L7 (HTTP/HTTPS)
	Access
	Direct pod access
	Smart routing
	Use case
	Internal/external exposure
	External HTTP routing
	________________


Simple analogy
Think of Kubernetes as a building:
      * Services = individual offices
      * Ingress = reception desk that directs visitors to correct office
      * Ingress Controller = receptionist following rules
________________


Key point
Ingress provides centralized, rule-based external HTTP/HTTPS routing into Kubernetes services.
________________


Interview-ready definition
Ingress in Kubernetes is a resource that defines rules for external HTTP and HTTPS traffic routing to internal services based on hostnames and paths, while an Ingress Controller implements these rules and manages the actual traffic routing within the cluster.
