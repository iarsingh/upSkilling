# job-application-bot

Playwright-driven job application automation for LinkedIn, Naukri, Indeed,
Greenhouse/Lever/Workday career pages, Uplers, and CutShort. Screening
questions and short free-text answers are generated by your local Ollama
model, grounded in `config/profile.json`.

## Read this before running

- **This submits real job applications with no human review step**, per your
  own configuration choice. A bug, a bad LLM answer, or a selector mismatch
  after a site redesign can send a bad application to a real employer. Run
  new/updated adapters with `--dry-run` first and check `data/applied-jobs.json`
  before trusting a site at volume.
- **LinkedIn's Terms of Service prohibit automating the platform**, and
  LinkedIn actively fingerprints and rate-limits automated browsing. This can
  get your account warned, feature-restricted, or suspended. That risk is
  yours to accept by using `--site linkedin`; nothing here defeats LinkedIn's
  detection, it just drives a real browser slowly and human-paced.
- **Uplers and CutShort adapters are unverified** (`src/sites/uplers.js`,
  `src/sites/cutshort.js`) - their selectors are best-effort guesses, not
  confirmed against the live site. Always `--dry-run` these first.
- Credentials are never stored in this repo. You log in by hand once per
  site in a real browser window; the session is saved to `browser-data/<site>/`
  (gitignored) and reused on every future run.

## Setup

```bash
cd job-application-bot
npm install
cp .env.example .env # edit if you want a different Ollama model/host
```

This drives your existing **Google Chrome** installation (`channel: "chrome"` in
`src/lib/browserSession.js`) rather than downloading Playwright's bundled
Chromium, so there's no ~150MB browser download step - just make sure Chrome
is installed at the usual `/Applications/Google Chrome.app` location.

Make sure Ollama is running locally (`ollama serve`) with a model pulled
(default `llama3.1:8b` - change `OLLAMA_MODEL` in `.env` to whatever you have).

Edit `config/profile.json` if your CTC, notice period, resume paths, or
target keywords change - it's the single source of truth every adapter reads
from.

## First run per site: log in

```bash
node src/cli.js --site linkedin --login-only
node src/cli.js --site naukri --login-only
node src/cli.js --site indeed --login-only
node src/cli.js --site uplers --login-only
node src/cli.js --site cutshort --login-only
```

Each opens a real Chromium window. Log in by hand (complete any 2FA/OTP).
The session is saved - you won't need to do this again until the site logs
you out. `generic-ats` needs no login (each application is a one-off public
form on the employer's own careers page).

## Running

```bash
# Dry run first - fills every form, stops before the final Submit click
node src/cli.js --site linkedin --keywords "DevOps Engineer Kubernetes GCP" --location "India" --limit 10 --dry-run

# Real run
node src/cli.js --site linkedin --keywords "DevOps Engineer Kubernetes GCP" --location "India" --limit 20

node src/cli.js --site naukri --keywords "Platform Engineer" --limit 30
node src/cli.js --site indeed --keywords "Site Reliability Engineer" --limit 15

# Company career pages (Greenhouse/Lever/Workday) - no search, just a URL list
node src/cli.js --site generic-ats --urls "https://boards.greenhouse.io/acme/jobs/123,https://jobs.lever.co/acme/456"
```

Add `--headless` once you've logged in once and trust the flow, to run
without a visible window.

## How it decides what to apply to

Every job's title + description is scored against `config/profile.json`'s
`preferredKeywords` and `targetRoles` via Ollama (`src/lib/ollama.js:scoreJob`),
falling back to a plain keyword-match heuristic if Ollama is unreachable.
Jobs scoring below the threshold are skipped and logged as such in the
console (not written to `data/applied-jobs.json`, so they won't block a
future re-run if you loosen the criteria).

## How form filling works

`src/lib/formFiller.js` matches each visible field's label/placeholder
against `config/profile.json` (name, email, CTC, notice period, experience,
etc.) first. If nothing matches - i.e. a genuinely open-ended screening
question like "Why do you want to join us?" - it falls back to Ollama
(`src/lib/ollama.js:answerScreeningQuestion`), grounded in your CV summary
and the specific job's description.

## Every application is logged

`data/applied-jobs.json` records every submitted (or attempted) application:
site, job title, company, URL, matched keywords, score, and status
(`submitted`, `dry_run`, or `incomplete_needs_review`). Already-logged jobs
are skipped on future runs (`jobKey` dedupe), so re-running the same search
won't double-apply. Delete entries from this file if you want to retry one.

## When a site changes its layout

Selectors live entirely in `src/sites/*.js` and `src/lib/formFiller.js`.
If a run starts logging a lot of `incomplete_needs_review`, the site likely
shipped a redesign - open the URL manually, inspect the new field labels/
button text, and update the relevant adapter. `--dry-run` is the fastest way
to confirm a fix without risking a bad real submission.

## Project layout

```
config/profile.json        Your CV summary, CTC, notice period, keywords, resume paths
src/lib/config.js           Loads profile.json + .env
src/lib/ollama.js           Local LLM: job scoring + screening-question answers
src/lib/browserSession.js   Persistent per-site Chromium profile + login wait
src/lib/applicationLog.js   data/applied-jobs.json read/write + dedupe
src/lib/formFiller.js       Label -> profile-value matching, LLM fallback
src/lib/genericFormPass.js  Site-agnostic "fill visible fields, click Next until Submit" loop
src/sites/linkedin.js       Easy Apply search + multi-step modal
src/sites/naukri.js         Quick Apply search + chatbot/form flow
src/sites/indeed.js         Native Indeed Apply + hands off external ATS redirects
src/sites/genericAts.js     Greenhouse / Lever / Workday direct-URL applications
src/sites/uplers.js         Experimental - unverified selectors
src/sites/cutshort.js       Experimental - unverified selectors
src/cli.js                  Entry point / argument parsing
data/applied-jobs.json      Application log (gitignored)
browser-data/<site>/        Saved login sessions (gitignored)
```
